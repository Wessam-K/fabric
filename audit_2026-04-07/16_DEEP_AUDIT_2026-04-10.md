## CRITICAL BLOCKERS (5) — Must Fix Before Release

### C-1: Password Policy Regressed to 6 Characters
**Severity**: CRITICAL | **Area**: Security | **File**: [backend/utils/validators.js#L74](backend/utils/validators.js#L74)

**What**: `MIN_PASSWORD_LENGTH = 6` with only a special character required. No uppercase, lowercase, or digit enforcement.

**Evidence**:
```javascript
const MIN_PASSWORD_LENGTH = 6;
function validatePassword(password) {
  if (!password || password.length < MIN_PASSWORD_LENGTH) { ... }
  if (!/[!@#$%^&*()_+\-=\[\]{};':"\\|,.<>\/?`~]/.test(password)) return '...';
  return null;
}
```

**Why it matters**: Passwords like `pass1!` or `abc12@` pass validation. Combined with a 10-attempt auth rate limit per 15min window, brute force is feasible. OWASP minimum is 8 characters; NIST recommends 12+.

**Root cause**: Likely a code revert or intentional relaxation that wasn't propagated to documentation.

**Fix**: Restore to min 10 chars + uppercase + lowercase + digit + special char.

---

### C-2: 2FA Backup Codes Stored in Plaintext
**Severity**: CRITICAL | **Area**: Security | **File**: [backend/routes/twofa.js#L34](backend/routes/twofa.js)

**What**: TOTP backup codes are stored as a plaintext JSON array in `users.totp_backup_codes`. No hashing applied.

**Evidence**: `db.prepare('UPDATE users SET totp_backup_codes = ? WHERE id = ?').run(JSON.stringify(backupCodes), user.id)`

**Why it matters**: If the database is compromised (file theft, SQL injection elsewhere, backup leak), all 2FA backup codes for all users are immediately usable. This defeats the purpose of 2FA entirely.

**Fix**: Store bcrypt hashes of each backup code. On verification, bcrypt.compare each code.

---

### C-3: Webhook URL SSRF — No Private IP Validation
**Severity**: CRITICAL | **Area**: Security | **File**: [backend/utils/webhooks.js#L107+](backend/utils/webhooks.js)

**What**: Webhook URLs are not validated against private/internal IP ranges. A user with webhook permissions can register `http://127.0.0.1:9002/api/...` or `http://169.254.169.254/latest/meta-data/` (AWS metadata) as a webhook target.

**Why it matters**: Server-Side Request Forgery (SSRF) allows attackers to:
- Access internal services not exposed to the internet
- Read cloud metadata credentials (AWS/GCP/Azure)
- Port-scan internal network
- Bypass firewalls

**Fix**: Validate webhook URLs against a blocklist of private IP ranges (RFC 1918, RFC 6598, link-local, loopback) before delivery. Use `new URL()` to parse, then DNS-resolve and check the IP.

---

### C-4: Unbounded Export Queries — Memory/DoS Risk
**Severity**: CRITICAL | **Area**: Performance/Availability | **File**: [backend/routes/exports.js](backend/routes/exports.js)

**What**: All 18+ export endpoints query ALL matching rows with no `LIMIT`. Date range defaults to 100 years if not specified.

**Evidence**: Every export handler runs `db.prepare('SELECT ... ORDER BY ...').all()` without row limits.

**Why it matters**: A single export request on a table with 100K+ rows could consume hundreds of MB of memory, crash the Node.js process, or lock the SQLite database for seconds. This is a denial-of-service vector.

**Affected endpoints**: `/api/exports/suppliers`, `/api/exports/fabric-usage`, `/api/exports/accessory-usage`, `/api/exports/wo-cost-breakdown`, `/api/exports/po-by-supplier`, `/api/exports/customers`, plus 12 more.

**Fix**: Add `LIMIT 10000` (configurable via env var) to all export queries. Return warning header if truncated.

---

### C-5: Backup System Has No Integrity Verification
**Severity**: CRITICAL | **Area**: Data Integrity | **File**: [backend/backup.js#L22](backend/backup.js)

**What**: Database backups are created via `fs.copyFileSync()` without any post-copy integrity check. A corrupted, truncated, or 0-byte backup will be stored without detection.

**Why it matters**: When disaster strikes and you restore from backup, you may find the backup is itself corrupted. This defeats the entire purpose of backups.

**Fix**: After copying, open the backup with better-sqlite3 in readonly mode and run `PRAGMA integrity_check`. Delete the backup if it fails.

## HIGH-PRIORITY ISSUES (14) — Should Fix Before Release

### H-1: WorkOrder Detail Executes 24 Sequential DB Queries
**Severity**: HIGH | **Area**: Performance | **File**: [backend/routes/workorders.js](backend/routes/workorders.js) (`getFullWO()`)

**What**: Every `GET /api/work-orders/:id` triggers 24 separate `db.prepare().get()/.all()` calls: main WO + fabrics + accessories + fabric_batches + accessories_detail + sizes + stages + extra_expenses + partial_invoices + cost_summary (7 more) + movement_log + consumption records + waste + invoices + aggregates.

**Impact**: Latency of 100-500ms per request depending on WO complexity. Under concurrent load, SQLite locking will serialize these, creating a bottleneck.

**Fix**: Consolidate into 3-4 queries using CTEs or batch `WHERE wo_id IN (?)`.

---

### H-2: PurchaseOrders Total Calculation Scans Entire Table
**Severity**: HIGH | **Area**: Performance | **File**: [backend/routes/purchaseorders.js#L30-L60](backend/routes/purchaseorders.js)

**What**: `GET /api/purchase-orders` loads ALL purchase orders into memory just to calculate summary statistics (totals by status), then paginates the actual response separately.

**Fix**: Replace with SQL aggregation: `SELECT status, SUM(total_amount) FROM purchase_orders GROUP BY status`.

---

### H-3: N+1 Query in WorkOrder Fabric Batch Creation
**Severity**: HIGH | **Area**: Performance | **File**: [backend/routes/workorders.js#L370-380](backend/routes/workorders.js)

**What**: When creating a work order with fabric batches, each batch is validated with a separate `SELECT * FROM fabric_inventory_batches WHERE id=?` query inside a loop.

**Fix**: Pre-load all batch IDs in one query: `WHERE id IN (${placeholders})`, then use a Map for O(1) lookups.

---

### H-4: User Import Has No Row Limit
**Severity**: HIGH | **Area**: Security/Performance | **File**: [backend/routes/users.js#L460-480](backend/routes/users.js)

**What**: `POST /api/users/import` processes unlimited rows. Each row runs `bcrypt.hashSync()` (expensive). 1,000 rows = 10+ seconds of blocking.

**Fix**: Reject imports exceeding 500 rows. Consider async processing for larger batches.

### H-5: Supplier Ledger Endpoint Returns Unbounded Data

**What**: `GET /api/suppliers/:id/ledger` fetches ALL purchase orders and ALL payments for a supplier with no pagination or LIMIT.

**Fix**: Add `LIMIT 1000` or pagination.

### H-6: Financial Reports Lack Pagination

### H-8: Stock Valuation Returns All Rows
**Severity**: HIGH | **Area**: Performance | **File**: [backend/routes/inventory.js#L180-220](backend/routes/inventory.js)

**What**: `GET /api/inventory/stock-valuation` returns ALL fabrics and accessories with stock, aggregating across all batches. No LIMIT.

### H-9: Logger Does Not Capture console.log Calls
**Severity**: HIGH | **Area**: Observability | **Files**: [backend/utils/logger.js](backend/utils/logger.js), 200+ console.error calls across routes

**What**: The Winston logger (`logger.js`) exists and has proper rotation/redaction, but it does NOT override `console.log/error/warn`. The 200+ `console.error()` calls in route files bypass the structured logger entirely — no rotation, no redaction of sensitive data, no JSON format.

**Fix**: Either (a) replace all `console.error()` calls with `logger.error()`, or (b) override global `console` methods to route through Winston.

---

### H-10: WebSocket Auth Bypass in Test/Dev Mode
**Severity**: HIGH | **Area**: Security | **File**: [backend/utils/websocket.js#L76](backend/utils/websocket.js)

**What**: In `test` and `development` NODE_ENV, WebSocket connections accept a plaintext `userId` from the client without JWT verification. If test mode is accidentally enabled in production, any client can impersonate any user.

**Fix**: Require JWT verification even in test mode. Use a test-specific JWT token instead.

---

### H-11: Reports Pivot Subquery Unbounded
**Severity**: HIGH | **Area**: Performance | **File**: [backend/routes/reports.js#L220-260](backend/routes/reports.js)

**What**: The `/api/reports/pivot` endpoint uses a subquery on `cost_snapshots` with `ROW_NUMBER() OVER (PARTITION BY wo_id ...)` that processes ALL rows before windowing, even though the outer query has `LIMIT`.

**Fix**: Add date range constraint or `LIMIT` to the inner subquery.

---

### H-12: No Database Integrity Check on Startup
**Severity**: HIGH | **Area**: Data Integrity | **File**: [backend/database.js](backend/database.js)

**What**: The database opens without `PRAGMA integrity_check`. If WAL corruption occurred during a crash, the application starts and silently operates on a corrupted database.

**Fix**: Run `PRAGMA quick_check` on startup (fast). If it fails, refuse to start and alert the operator.

---

### H-13: No Request Timeout on Express
**Severity**: HIGH | **Area**: Availability | **File**: [backend/server.js](backend/server.js)

**What**: No global request timeout. A slow query or stuck middleware can hold a connection open indefinitely, eventually exhausting available connections.

**Fix**: Add `server.setTimeout(30000)` or use `connect-timeout` middleware.

---

### H-14: Nginx Proxy Has No Timeout Configuration
**Severity**: HIGH | **Area**: Availability | **File**: [nginx.conf](nginx.conf)

**What**: `proxy_pass` directives have no `proxy_read_timeout`, `proxy_connect_timeout`, or `proxy_send_timeout`. A stuck backend request will keep the nginx connection open indefinitely.

**Fix**: Add `proxy_read_timeout 30s; proxy_connect_timeout 5s; proxy_send_timeout 30s;` to each location block.

## Performance & Scalability Review

### Current Bottlenecks
1. **Single SQLite connection** — All requests serialized through one connection. Under 50+ concurrent users, this becomes the bottleneck.
2. **24-query getFullWO()** — Most expensive endpoint, called on every work order view.
3. **Full-table-scan totals** — PurchaseOrders, Suppliers, Inventory recalculate totals by loading all rows.
4. **Unbounded exports** — Can load entire tables into memory.
5. **No caching** — Every request hits the database. No Redis or in-memory cache for hot data.

### Scalability Limits
- **SQLite ceiling**: ~100 concurrent read requests and ~1 write at a time. Adequate for 20-50 users.
- **No horizontal scaling**: Single-server architecture. Docker containers run one backend instance.
- **No CDN**: Static assets served directly from Express/nginx.

### What Will Break First
Under real traffic (50+ users), the first things to fail:
1. Work order detail page (24 queries)
2. Export downloads (memory)
3. Report generation (unbounded queries)

### What's Excellent

- Code organization is **clean**: clear separation of routes, middleware, utilities
- Arabic RTL support is **complete** and consistent

### What Needs Work
- Several security items regressed or were missed (password policy, 2FA storage, SSRF)
- Performance under load is **not validated** — multiple unbounded queries will fail at scale
- Observability is **incomplete** — structured logger exists but isn't used uniformly
- Error feedback to users is **poor** in many edge cases
- Backup system is **incomplete** — no integrity verification

### Bottom Line
The application is well-engineered with strong fundamentals. The 5 critical issues are all **low-effort fixes** (1-2 hours total). After those fixes, the app is safe to ship for a small to medium user base (up to ~30 concurrent users). The performance optimizations (H-1 through H-8) should follow shortly after to support growth.
